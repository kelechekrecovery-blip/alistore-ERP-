## 1. Hardening (продакшн-готовность) — высокий приоритет

1. **Rate limiting** на публичных/абьюз-опасных эндпоинтах: `POST /checkout`, OTP-выдача,
   `POST /support/tickets`, `POST /payments/*/webhook`. NestJS `@nestjs/throttler`.
   - Приёмка: превышение лимита → 429; нормальный трафик не задет; тест на лимит.
2. **E2E-набор (Playwright)** для 5 критичных потоков: покупка web (cart→checkout→order),
   POS-продажа со скидкой→approval, возврат→refund, обмен, скупка Б/У intake.
   - Приёмка: `playwright test` зелёный в CI; артефакты (screenshot/video) на падении.
3. **CI-пайплайн** (GitHub Actions): install → lint → typecheck → `api:test` → `next build`.
   - Приёмка: PR блокируется красным пайплайном; кэш зависимостей.
4. **Чистый воспроизводимый seed.** Dev-БД замусорена QA-прогонами + демо-правками Claude
   (overstock SGS-24, backdate платежей 06-27, IMEI-коллизия IPH-15-128-UNIT-1). Нужен
   детерминированный seed без этого мусора + скрипт полного reset.
   - Приёмка: `db:reset && db:seed` даёт чистый предсказуемый набор; loss-prevention сигналы
     пусты на чистом seed (кроме намеренно заложенных демо-кейсов, если нужны).
5. ✅ **Guard покрытия Event Ledger (инв #10)** — СДЕЛАНО (Claude, субагент):
   `apps/api/test/ledger-coverage.e2e-spec.ts` доказывает co-commit `payment.received` +
   `unit.sold` на реальной POS-продаже. Можно расширить на refund/status-переходы при желании.

## 2. Добор фич — средний приоритет

6. **Admin Product Management UI** (CRUD товаров + approval на опасные правки). **Важно:**
   это UI-дом для готовых keyless-AI эндпоинтов Claude — впаять кнопки
   «Авто-категория» (`POST /ai/categorize`) и «Сгенерировать описание» (`POST /ai/describe`)
   в форму товара. Изменение цены/архив — через approval (уже есть executors).
   - Приёмка: создать/править/архивировать товар из UI; авто-категория и описание
     подставляются одним кликом; опасные правки паркуются в Approval Inbox.
7. **Notification-шаблоны** на все транзакционные события (заказ подтверждён / готов к выдаче /
   гарантия обновлена / напоминание о долге) через Novu/outbox.
   - Приёмка: каждое событие рендерит шаблон с корректными полями; consent-фильтр применён.
8. **Подарочные карты / стор-кредит** (Phase 12, без внешних зависимостей): выпуск, баланс,
   списание в checkout/POS, ledger-события.
   - Приёмка: выпуск карты → оплата ею в checkout уменьшает баланс; двойное списание невозможно.
