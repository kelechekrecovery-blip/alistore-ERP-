## А. Как устроены деньги в системе (1 страница, по файлам)

- **Event Ledger** — каждая значимая операция пишет append-only событие `AuditEvent`
  (каталог: `apps/api/src/audit/event-types.ts`). Отчёты, риски и финансы читают из
  одного журнала, поэтому цифры не расходятся. Событие и бизнес-запись коммитятся
  атомарно, дубликаты применяются один раз (idempotency).
- **Двойная запись** — `apps/api/src/finance/accounting-journal.ts`: проводка требует
  минимум 2 строки, дебет = кредит, иначе отказ (`accounting_entry_unbalanced`);
  один первичный документ → одна проводка; счета валидируются по плану счетов.
- **План счетов (рабочий, внутренний)** — `apps/api/src/finance/accounting-chart.ts`:
  1000 касса, 1010 р/с, 1020 деньги у платёжных провайдеров, 1100 дебиторка,
  1200 запасы, 1250 подотчёт, 1300 авансы поставщикам, 1400/1410 ОС и амортизация,
  2000 поставщики, 2100 зарплата, 2200 налоги, 2300 подарочные карты, 3000 капитал,
  4000/4100 выручка розницы/сервиса, 5000 себестоимость, 6100–6990 расходы.
  ⚠️ Это внутренний план, а не утверждённый план счетов КР — вопрос №1 бухгалтеру.
- **Платежи маппятся автоматически** (`accounting-journal.ts:46–50, 52–123`): наличные →
  1000, подарочная карта → 2300, всё прочее (карта, QR, рассрочка) → 1020; выручка
  4000 (сервис 4100), налог выделяется «в том числе» на 2200.
- **Налоги** — `apps/api/src/finance/sales-tax.ts`: налог считается «включённым»
  (included tax), ставка в базисных пунктах (bps), скидки распределяются по позициям,
  округление поглощается последней позицией. Код НЕ знает реальных ставок КР:
  по умолчанию товар создаётся с `taxCode='vat_standard'`, `taxRateBps=1200`
  (`apps/api/src/products/products.service.ts:132–133`) — **это неподтверждённый
  дефолт, утвердить или заменить (вопрос №2)**.
- **Закрытие периода** — мягкое (`soft_closed`: только налоговая сверка и сторно) и
  жёсткое (`hard_closed`: ничего) (`accounting-journal.ts:194–215`). Готовность периода
  проверяется 8 блокерами (`apps/api/src/finance/finance.service.ts:2185–2264`).
- **FX-гвард** — курсы валют к KGS (`AccountingCurrencyRate`, микро-единицы), отчёт
  переоценки валютных расходов (valuation delta, пропуски курсов), а незакрытая
  валютная переоценка **блокирует** закрытие периода (`fx_revaluation_required`).
- **Фискализация отсутствует** — чеки печатаются с пометкой «Информационный чек —
  фискализация не выполнена» (`apps/api/src/receipts/receipts.service.ts:89`),
  провайдер-заглушка `INFORMATIONAL_FISCAL_PROVIDER` (`apps/api/src/fiscal/fiscal-provider.ts`).
  ККМ/ОФД КР — отдельный гейт `GAP-FISCAL-001`, `docs/GO-LIVE-PHASES.md` G3.

## В. Что система УЖЕ умеет (со ссылками)

- **Готовность закрытия периода — 8 блокеров** (`finance.service.ts:2185–2264`,
  API `GET /finance/periods/:period/readiness`): незакрытые финсверки, открытые
  кассовые смены, незавершённые возвраты, незакрытые счета поставщиков, несверенные
  банковские выписки, невыплаченные payroll-запуски, открытые подотчётные авансы,
  валютные расходы без FX-переоценки. Мягкое/жёсткое закрытие — `accounting-journal.ts:194–215`.
- **AR aging** (дебиторка клиентов по корзинам): `finance.service.ts:952`
  (`customerAging`), drill-down по клиенту — `:1092`; AP aging по поставщикам — `:917`.
- **Payroll accrual из табеля**: расчёт из смен/attendance + комиссия от выручки,
  идемпотентное проведение (один запуск на период×точку), выплата только с номером
  платёжного документа — `apps/api/src/hr/hr.service.ts:69–186`.
- **Налоговая сверка периода** (settle tax period) и сторно в мягко закрытом периоде —
  `accounting-journal.ts:204–215`, `SettleTaxPeriodDto`.
- **FX-переоценка валютных расходов** с дельтой к текущему курсу и детекцией пропусков
  курсов — `finance.service.ts:536–619`.
- **Банковские выписки**: импорт периода, статус reconciled, блокируют закрытие периода —
  `finance.service.ts:693+`, `bank-statement.controller.ts`.
- **ОС и амортизация**: помесячная последовательная амортизация с проводкой Д6700 К1410 —
  `finance.service.ts:347–397`.
- **Подотчёт**: авансы, закрытие (отчёт/возврат), лимиты остатка — `finance.service.ts:225–250`.
- **Инкассация**: только после закрытия и пересчёта смены, с остатком к инкассации —
  `finance.service.ts:643–663`.
- **Расчёт налога «в том числе»** с распределением скидок и корректным округлением при
  сплит-тендерах — `apps/api/src/finance/sales-tax.ts`.
- **Консигнация** полный цикл: приём, продажа, выплаты комитенту, возвраты, корректировки —
  события `consignment.*` (`event-types.ts:32–37`).
- **Append-only аудит** всех денежных операций с акторами и связями — `AuditEvent`,
  отчёты читают только отсюда.
