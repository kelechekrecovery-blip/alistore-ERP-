# AliStore Web Audit

Date: 2026-07-19
Scope: local sandbox/demo contour for the Web, ERP and ERP-to-storefront contract.

## Result

The audited Web surface is functional for the covered sandbox flows. The local
production build, API integration suites, route inventory, ERP CMS flows and
ERP product administration pass on the current source tree. This report does
not certify live providers, a durable cloud deployment, physical devices or
production financial operations.

## Gates

| Gate | Result | Evidence |

| API build | PASS | `npm run api:build` |
| Web production build | PASS | 43 generated routes in `mvp:verify` |
| API integration | PASS | 172/172 isolated Jest files |
| Full browser run | PASS | 112/112 on a cold isolated API/Web pair |
| ERP secure + CMS regression | PASS | 7/7 on isolated API/Web servers |
| Web route audit | PASS | 46/46 on a cold isolated API/Web pair |

An earlier full browser run found five non-functional CMS locator failures:
the CMS button exposed only one of two established names. The accessible name
now includes both `Админка сайта` and `Сайт · CMS витрины`; the focused ERP/CMS
suite passed 7/7 and the corrected full browser suite passed 112/112.

## Route coverage

The route audit covered the following groups:

- Public: `/`, `/about`, `/catalog`, `/search`, `/compare`, `/favorites`, `/cart`, `/checkout`, `/login`.
- Service and legal: `/delivery`, `/oferta`, `/privacy`, `/support`, `/trade-in`, `/exchange`, `/warranty`, `/assess`, `/b2b`, `/app`, `/tg`.
- Invalid/deep-link safety: product, order, account order/status and account warranty identifiers return a non-crashing not-found state without a predictable invalid-ID API error.
- Customer and protected shells: account, addresses, bonuses, devices, notifications, protection, returns, settings, admin products, AI tools, approvals, ERP, POS, refunds, Staff and Warehouse.
- Runtime and indexing: `/healthz`, Apple association, Android asset links, `/robots.txt` and `/sitemap.xml`.

All 46 route cases passed. Apple/Android association endpoints are accepted as
expected sandbox responses when signing identifiers are not configured; real
domain association remains a release gate.

## ERP to storefront coverage

The accepted browser flows cover product publication, price approval, CMS
draft/publish/reorder, review moderation, promotion quote redemption, delivery
point and slot availability, checkout, return/refund, procurement-to-sale,
warehouse stock and service/customer visibility. The API and Event Ledger
remain authoritative in these scenarios; browser storage and search are not
used as business truth.

## Changes made

- Added the repeatable `web:route-audit` Playwright inventory with console,
  failed-request, HTTP-error, empty-page and horizontal-overflow checks.
- Isolated E2E Next build directories per web port so parallel local runs do
  not share a `.next` lock or cache.
- Prevented invalid product deep links from issuing a predictable malformed
  catalog request; the page now resolves to its not-found state cleanly.
- Stabilized the ERP CMS accessible-name contract for desktop and mobile
  navigation.

## Remaining release blockers

- Full cross-browser, accessibility and performance evidence still needs a
  dedicated release run.
- `ecosystem:audit` is affected by unrelated parallel iOS asset changes in the
  current worktree and is not claimed green here.
- The public site remains a demo/sandbox contour; live payment, SMS, fiscal,
  push, R2 evidence, Render staging, backup/restore and physical-device gates
  remain external.
- Missing design references must be restored or explicitly retired by the
  owner; they are not fabricated by this audit.
