# AliStore Web Audit

Date: 2026-07-19
Scope: local sandbox/demo contour for the Web, ERP and ERP-to-storefront contract.

## Result

The audited Web surface is functional for the covered sandbox flows. The local
production build, API integration suites, route inventory, ERP CMS flows and
ERP product administration pass on the current source tree. This report does
not certify live providers, a durable cloud deployment, physical devices or
production financial operations.

### Findings confirmed by source inspection

- **Production payment activation is not complete.** The production gateway
  adapter is deliberately fail-closed and raises
  `production_payment_gateway_not_activated` until a provider contract and
  signed implementation are supplied. This is an external/provider blocker,
  not a Web UI bug.
- **Sandbox confirmation is explicitly gated.** The sandbox webhook is only
  reachable when the sandbox provider and `PAYMENTS_SANDBOX_CONFIRM_ENABLED`
  are both enabled. It must remain disabled on any real-traffic deployment.
- **OTP echo is fail-closed in production source.** `devCode` is returned only
  when `AUTH_OTP_DEV_ECHO=true` and `NODE_ENV` is not `production`; deployment
  readiness still has to be run after redeploy.
- **Browser sessions remain in `localStorage`.** This is a release-hardening
  issue: migrate access/refresh tokens to an HttpOnly, Secure, SameSite cookie
  design with CSRF protection before accepting sensitive production traffic.
- **Live deployment cannot be certified from local tests.** The public API
  needs redeployment of the current source, then a post-deploy check for
  Swagger 404, health, CORS, host allowlist, demo mode and provider readiness.

The local Web surface is therefore **functional for the covered sandbox
flows**, but the project is not production-ready: the trusted audit lock,
cookie session design, provider activation, durable deployment, real
credentials, R2 Evidence, backup/restore and physical-device gates remain
open.

The COD handover contract now checks receivable coverage for delivered orders,

Remaining Web release risks: browser tokens are stored in `localStorage`,
production payment activation is deliberately fail-closed, Render/origin
deployment is unavailable, and the trusted ecosystem audit is blocked by its
stale package-lock hash.

The customer Web auth slice was then migrated to an HttpOnly cookie session:
the browser keeps only the short-lived access token in memory, refresh rotates
inside a protected cookie, legacy customer token storage is cleared on mount,
and native bearer-token clients remain compatible. HTTP smoke confirmed OTP
verification, cookie-authenticated `/auth/me`, refresh rotation and the absence
of `refreshToken` in Web JSON. Staff/ERP still require a separate cookie-session
migration before this release risk can be closed globally.

### Recheck 2026-07-21C

- The first cookie-session recheck exposed an isolated E2E configuration gap: the Playwright API server had no `CORS_ORIGINS`, so credentialed browser requests were rejected by preflight. The E2E server now declares both local Web origins without weakening production policy.
- The next run exposed an expected anonymous `422 refresh_invalid` as a route-audit HTTP error. Web login now also sets a root-scoped, non-secret `alistore_session_hint`; anonymous mounts skip refresh, while authenticated mounts still rotate the HttpOnly refresh cookie. Logout clears the hint.
- Web session contract: PASS `4/4`.
- Web unit tests: PASS `68/68`.
- Isolated Chromium route audit: PASS `46/46` in `3.1m`.
- This closes the local route-audit regression. It does not close Staff/ERP browser token storage, public HTTP 530, live providers, staging, or native physical-device gates.

- Staff Web integration: PASS `4/4`; Staff Chromium UI: PASS `2/2`; Web unit: PASS `68/68`; API build and Web production build: PASS; route audit: PASS `46/46`.
