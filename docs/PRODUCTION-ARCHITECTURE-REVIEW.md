# Почему ali.kg постоянно падает — разбор фактической архитектуры

Дата: 2026-07-22. Повод: «сайт много времени не работает».

## Что происходит на самом деле

Боевой сайт обслуживается **не облаком, а этим рабочим ноутбуком** через Cloudflare Tunnel:

```
покупатель → Cloudflare → cloudflared tunnel (процесс на Mac)
                              ├── ali.kg      → localhost:3000  (Next.js витрина)
                              ├── admin.ali.kg→ localhost:3000
                              └── api.ali.kg  → localhost:4000  (NestJS API)
                                                     └── Postgres alistore_dev (локальная БД на Mac)
```

Установлено фактически, не по документации:
- запущен процесс `cloudflared tunnel run` на этой машине;
- на портах 3000/4000 слушают локальные Node-процессы;
- `https://api.ali.kg/api/health/ready` отдаёт ровно то же, что `http://127.0.0.1:4000/...`;
- локальная база `alistore_dev` содержит **те же 18 товаров**, что видит покупатель на ali.kg.

## Почему это и есть причина простоев

| Событие | Что видит покупатель |
|---|---|
| Ноутбук заснул / закрыли крышку | Cloudflare error 1033 → **530** |
| Упал/не запущен процесс витрины (3000) | **502** (именно это и было сейчас) |
| Перезагрузка, обновление, смена сети | сайт недоступен |
| Ноутбук выключен | магазина нет |

Это не баг в коде. Код исправен: прод-сборка витрины отдаёт 200 на всех ключевых
страницах, бэкенд-гейт зелёный (957 тестов). Падает **способ размещения**.

## Ещё одна находка: боевой API работает в dev-режиме

Процесс на 4000 запущен как `ts-node src/main.ts` — это **режим разработки**
(TypeScript компилируется на лету), а не собранный `dist/main.js`. Медленнее,
прожорливее по памяти и не то, для чего предназначен прод. Собранный `dist`
при этом есть — то есть боевой трафик обслуживает dev-runner просто потому, что
так когда-то запустили вручную.

Не перезапускал: сейчас всё работает, а рестарт живого API ради режима — риск
без срочной пользы. При переезде на Render это уходит само (там docker-образ
запускает `dist`).

## Риск, который важнее простоя: данные

Боевые данные лежат в **локальной dev-базе на ноутбуке** (`alistore_dev`), без
управляемых бэкапов и репликации. Пока заказов 0 — потерь нет. Но как только
пойдут реальные продажи, вся выручка, клиенты и движения кассы будут храниться
на одном диске без резервного копирования. Отказ диска = потеря всего учёта.

**Немедленная мера, пока не переехали:** регулярный дамп
`pg_dump alistore_dev > backup-$(date +%F).sql` и копия вне ноутбука.

## Куда переезжать (уже описано в репозитории)

`render.yaml` описывает полноценный боевой стек — он просто не развёрнут:

- `alistore-api-prod` (web, docker, healthcheck `/api/health/ready`)
- `alistore-web-prod` (web, docker, healthcheck `/healthz`)
- `alistore-worker-prod` (фоновые задачи)
- `alistore-backup-prod` (cron-бэкапы)
- `alistore-meilisearch-prod` (поиск)
- управляемые Postgres + Redis

То есть архитектура «как надо» уже спроектирована и лежит в репозитории. Не хватает
только развёртывания и переключения DNS.

## План перехода

1. **Развернуть Blueprint** в Render (`render.yaml`) — 5 сервисов + managed Postgres/Redis.
2. **Перенести данные**: `pg_dump` локальной базы → восстановить в managed Postgres.
   Не оставлять имя `_dev` для боевой базы.
3. **Задать секреты** в Render (они `sync: false`, вводит владелец).
4. **Переключить DNS/Cloudflare** с туннеля на Render-origin для `ali.kg`,
   `admin.ali.kg`, `api.ali.kg`.
5. **Погасить туннель** — он больше не нужен; заодно исчезнет зависимость от ноутбука.
6. Проверить: health обоих сервисов, каталог, карточка товара, чекаут.

После этого сайт перестаёт зависеть от того, включён ли ноутбук.

## Временная мера, если переезд не сегодня

Пока живём на туннеле — как минимум:
- не давать Mac засыпать (**`sudo pmset -a disablesleep 1`**, см. уточнение ниже);
- автоперезапуск процессов (launchd-агент), чтобы падение 3000/4000 не оставляло
  магазин лежать до ручного вмешательства;
- ежедневный `pg_dump` наружу.

Это снижает частоту простоев, но не убирает причину: боевой магазин не должен
жить на рабочей машине.

# Дополнение 23.07.2026 — измеренная причина и что сделано

Повод: «сайт постоянно лежит». Разбор выше был верен по сути, но одна его
рекомендация не работает, а часть причин не была видна. Ниже — уточнение.

## Причина названа точно: сон от закрытой крышки

Из `pmset -g log` за 20–23 июля:

| Заснул (Clamshell Sleep) | Проснулся | Простой |
|---|---|---|
| 20.07 19:24 | 20.07 23:25 | 4ч 01м |
| 21.07 00:47 | 21.07 09:37 | 8ч 50м |
| 21.07 19:27 | 21.07 19:50 | 23м |
| 21.07 21:35 | 21.07 22:21 | 46м |
| 22.07 04:51 | 22.07 11:52 | 7ч 01м |
| 22.07 19:13 | 22.07 20:16 | 1ч 03м |
| 22.07 22:23 | 22.07 22:47 | 24м |
| + 3 коротких | | ~10м |

**22ч 37м простоя из 52ч → доступность 57%.** Просыпается машина только от
касания (`RTP.multi-touch/HID Activity`): магазин работает ровно тогда, когда
владелец сидит за ноутбуком.

## Опровержение: `caffeinate` эту задачу не решает

Рекомендация выше (`caffeinate -dims`) **не защищает от сна по крышке** — она
подавляет только *idle*-сон. Доказательство на этой машине: агент
`com.codex.keepawake` работал с 11:52, а `Clamshell Sleep` в 22:23 всё равно
произошёл. Единственная рабочая мера — `sudo pmset -a disablesleep 1`
(сохраняется между перезагрузками; обязательны питание от сети и вентиляция).

## Чего не было видно раньше

1. **Боевой API был дочерним процессом приложения ChatGPT/Codex**
   (`59978 → 59918 → 59888 → 14785`): перезапуск Codex убивал магазин.
   Витрина работала сиротой с `PPID 1`. Супервизора не было ни у одной.
2. **Сторож `keep-site-up.sh` был написан, но не установлен** — в
   `~/Library/LaunchAgents/` лежал только `com.alistore.cloudflared`.
3. **Dev-режим отключал боевые guard'ы.** Проверено запросами к живому API:
   `CORS` отражал любой `Origin`; `https://api.ali.kg/api/docs` отдавал `200`
   (полная схема внутреннего API наружу); `trust proxy` был выключен, поэтому за
   cloudflared `req.ip` равнялся `127.0.0.1` у всех клиентов и все `@Throttle`-
   эндпоинты делили один бакет — один человек гасил вход по SMS всем; cookie
   сессий шли без флага `Secure`.
4. **Бэкапы молча не работали.** `launchctl print` показывал `runs = 0`,
   последний дамп отставал на два дня: задание стояло на 03:17, когда машина
   спала, при `RunAtLoad=false`. Все копии лежали на диске самой базы.

## Что исправлено

- `CORS_ORIGINS` и `TRUST_PROXY_HOPS` заданы в `apps/api/.env`. Обе переменные
  действуют независимо от `NODE_ENV`, поэтому дыры закрыты сразу. Число хопов
  выверено опытом: запрос снаружи через свободный ingress `new.ali.kg` показал
  `X-Forwarded-For` ровно с одной записью — реальным IP клиента.
- API переведён с `ts-node` на собранный `dist/main.js`.
- Написаны `scripts/com.alistore.api.plist`, `scripts/com.alistore.web.plist` и
  обёртки `scripts/run-api-prod.sh`, `scripts/run-web-prod.sh`.
- `scripts/keep-site-up.sh` переписан: вместо `nohup`-сирот (именно они и
  породили нынешний бардак) — `launchctl kickstart` существующего агента.
- Бэкапы: `RunAtLoad=true`, второе окно 13:17, выгрузка копии в iCloud Drive,
  проверка размера дампа. Восстановление проверено: 131 таблица, ноль
  предупреждений, счётчики боевой и восстановленной БД совпали.

## Остаётся нерешённым

- **Автоперезапуск не включён: TCC.** macOS запрещает launchd-агентам доступ к
  `~/Desktop`, где лежит репозиторий. Проверено пробным агентом: корень домашней
  папки — `OK`, `Desktop` и `Documents` — `DENIED`. Агенты падают с кодом 126.
  Лечится выдачей Full Disk Access для `/bin/bash` либо переносом репозитория
  за пределы `~/Desktop`.
- **`NODE_ENV=production` не включён.** `main.ts:13` вызывает
  `assertProductionRuntimeReady`, который валит старт, пока не зелены все
  проверки `health/production-preflight.ts`. Блокируют 8, и каждая требует
  инфраструктуры: алерты в Telegram, S3 для медиа (сейчас паспорта раздаёт
  локальный диск без подписи), транспорт уведомлений, BullMQ/Redis, флаги
  outbox/sweep/reminders/refund-relay. Заготовка с разбором — в шапке
  `apps/api/.env.production`; актуальный список — `npm run launch:preflight`.
  Поэтому остаются открытыми `/api/docs`, отсутствие HSTS, cookie без `Secure`
  и Host-allowlist на уровне приложения.
- **Мониторинга нет.** О простое по-прежнему узнают, открыв сайт.
- **Главное — зависимость от одного ноутбука.** Всё выше сокращает простой, но
  не убирает причину. План переезда — в разделах выше.
