# Спайк: изоляция API-тестов (VERIFY-078 / VERIFY-057)

**Статус:** дизайн-спайк, реализация не начата. Один вертикальный срез плана
ruflo-харденинга (Фаза 4). Цель — снять правило «один прогон API-тестов на машину»,
из-за которого красный результат неотличим от дефекта при параллельном Codex/CI.

## Проблема (сверено по коду)

Все `apps/api` jest-сьюты делят **одну** БД `alistore_test` и чистят фикстуры голым
`deleteMany()`: **1175 вызовов в 108 спек-файлах, 72 из них стирают `AuditEvent`
целиком**, плюс глобальный `TRUNCATE` в `test/db-test-cleanup.ts` на каждый `beforeEach`.
Пока это так, **ни один прогон не воспроизводим при втором работающем процессе**: чужой
`deleteMany` попадает между `findFirst` и `FOR UPDATE` и роняет посторонние сьюты
(проверено: коммит давал 18/18 в одиночку и 2 FAIL под конфликтом). Уже стоило ложных
дефектов: `VERIFY-057/075/077`, `OWNER-AUDIT-003`, четыре сьюта 21.07 — одна причина.

Оба гейта завязаны на это: CI `npm run api:test` (один процесс, `--runInBand`) и
`mvp:verify` батчер (`run-api-test-batches.mjs`, один спек-файл на процесс, 175 процессов).
Батчер уже платит за изоляцию временем (оценивал работу в 4504с против 78с у CI).

## Два направления

### A. Схема-на-процесс (`?schema=test_<worker>`) — рекомендую
Каждый jest-процесс/worker работает в своей Postgres-схеме. `search_path` изолирует
таблицы, `deleteMany`/`TRUNCATE` остаются как есть, но не пересекаются между процессами.

**Механизм:**
- `test/setup-env.ts`: вычислить суффикс схемы из `process.env.JEST_WORKER_ID` (или pid)
  и переписать `TEST_DATABASE_URL`/`DATABASE_URL` на `...?schema=test_w${id}`.
- `test/setup-db.ts` `beforeAll`: `CREATE SCHEMA IF NOT EXISTS`, применить схему
  (`prisma db push --skip-generate` в эту схему **один раз на процесс**, кэшировать по
  имени схемы), `afterAll`/глобальный teardown: `DROP SCHEMA ... CASCADE`.
- Prisma читает схему из `?schema=` в URL — отдельный `PrismaClient` на процесс уже так и есть.

**Плюсы:** полная параллельная безопасность; `deleteMany`/`TRUNCATE` не трогаем (0 правок в
108 файлах); снимает правило «один прогон на машину»; CI и локально можно `--maxWorkers>1`.
**Минусы:** `db push` на схему стоит времени; при N процессах — N пушей (кэш по схеме
амортизирует внутри процесса, но батчер стартует новый процесс на спек → 175 пушей). Нужен
надёжный teardown, иначе схемы копятся (митигация: префикс `test_w*` + drop-all в pretest).
**Стоимость:** ~2 файла харнеса (`setup-env`, `setup-db`) + npm-хук на префиксную очистку.
Риск на 108 файлов — нулевой (их не трогаем).

### B. Транзакционный откат (тест в транзакции, ROLLBACK после)
Каждый тест оборачивается в транзакцию, откатываемую в `afterEach`; фикстуры исчезают без
`deleteMany`.

**Почему плохо ложится на этот код:** приложение массово использует **настоящие**
`audit.transaction` (`prisma.$transaction`), advisory-locks (`pg_advisory_xact_lock`) и
`FOR UPDATE`. Обёртывающая тест-транзакция даёт **вложенные** транзакции, которых Prisma
нормально не поддерживает; advisory-xact-locks и `FOR UPDATE` внутри внешней tx ведут себя
иначе (лок держится до конца внешней, а не внутренней) — это меняет ровно те инварианты,
которые сьюты и проверяют (idempotency/replay/concurrency). Плюс `PrismaService` инжектится
напрямую в сервисы — пришлось бы подменять DI на per-test tx-клиент во всём графе.
**Вывод:** для леджер-архитектуры этот путь ломает семантику под тестом; не рекомендую.

## Рекомендация

**Схема-на-процесс (A).** Она не трогает 108 файлов с `deleteMany`, изолирует по процессам
(ровно тот параллелизм, что сейчас запрещён), и совместима с обоими гейтами. B отпадает
из-за вложенных транзакций и advisory-lock/FOR-UPDATE семантики — они центральны для этого
леджера.

## План раскатки (отдельными срезами, каждый с гейтом)

1. **PoC** ✅ *механизм проверен на `alistore_test`*: две схемы `test_poc_w1/w2` с
   одноимённой таблицей; `TRUNCATE` в w1 (аналог `deleteMany`) оставил данные w2
   нетронутыми (w1→0 строк, w2→1), схемы дропнуты `CASCADE` чисто. Изоляция по схемам
   работает. **Остаётся замерить** стоимость `prisma db push` на новую схему (100 миграций)
   — это решает выбор «схема на спек-процесс» vs «схема на worker».
2. **Харнес** (1 срез): `setup-env` вычисляет схему из `JEST_WORKER_ID`/pid; `setup-db`
   create/push(cached)/drop; глобальный `pretest` дропает висячие `test_w*`.
3. **Параллельный прогон** (1 срез): снять `maxWorkers: 1` в `jest.config.js`, замерить
   стабильность на `--maxWorkers=50%`; обновить `run-api-test-batches.mjs` (батчинг больше
   не нужен для изоляции — можно вернуть `API_JEST_BATCH_SIZE` к разумному или убрать).
4. **Снять правило** из `CLAUDE.md` («один прогон на машину») и `BACKLOG` VERIFY-078/057.

## Риски / открытое

- Стоимость `db push` на схему при 175 процессах — измерить в PoC; если дорого, перейти на
  worker-schema (`--maxWorkers`, схема на worker, а не на спек-процесс) — тогда пушей единицы.
- Teardown-надёжность: упавший процесс оставит схему; префиксная drop-all в `pretest` обязателен.
- Взаимодействие с `mvp:verify` guardrails (имя БД должно содержать `test`) — суффикс схемы
  не меняет имя БД, инвариант сохраняется.
- Гейт `check-no-fixtures` и migration-upgrade тесты — не затрагиваются (другой слой).

## Что НЕ делаем в этом спайке
Правку 108 файлов, снятие `deleteMany`, изменение `jest.config`/батчера — это шаги 2–4 выше,
отдельными срезами после PoC. Спайк только фиксирует направление и механизм.
