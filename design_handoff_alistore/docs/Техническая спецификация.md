# AliStore — Техническая спецификация (production-grade)

Цель: система, которая держит рост от одного магазина до сети + маркетплейса,
не теряет деньги, не путает склад и переживает реальную эксплуатацию.

---

## 1. Архитектура (высокоуровневая)

```
Клиенты: Web(Next.js) · iOS(Swift) · Android(Kotlin) · POS · ERP(web) · Telegram
        │
   API Gateway (auth, rate-limit, versioning /v1)
        │
   Backend (NestJS, модульный монолит → выделяем в сервисы по нагрузке)
   ├─ Orders / Payments / Inventory / IMEI / Cash / Courier
   ├─ Returns / Warranty / Supplier / Pricing / Approval
   ├─ CRM / Campaigns / Notifications / Support
   └─ AI-слой (оценка Б/У, цены, ассистент) — отдельный сервис
        │
   Data: PostgreSQL(основная) · Redis(кэш/очереди/локи) ·
         Elasticsearch/Meilisearch(поиск) · S3/MinIO(файлы) ·
         ClickHouse(аналитика, опц.)
        │
   Event Bus (Kafka/NATS/Redis Streams) → Event Ledger (append-only)
```

**Принцип:** старт как **модульный монолит** (быстро, дёшево), с чёткими границами
модулей, чтобы под нагрузкой безболезненно вынести горячие части (Orders, Payments,
Search, Notifications) в отдельные сервисы. Не начинать с микросервисов — это преждевременно.

---

## 2. Стек (обязательный)

**Backend:** Node 20 LTS, NestJS, TypeScript strict, Prisma ORM, PostgreSQL 15+.
**Web:** Next.js 14 App Router, React, TS, Tailwind, TanStack Query, Zod.
**iOS:** Swift 5.9, SwiftUI, Swift Concurrency, Core Data (офлайн), Keychain.
**Android:** Kotlin, Jetpack Compose, Coroutines/Flow, Room (офлайн), DataStore.
**Инфра:** Docker, Kubernetes (или Nomad), Nginx/Traefik, CDN (Cloudflare).
**CI/CD:** GitHub Actions → build/test/lint → staging → prod (blue-green/canary).
**Наблюдаемость:** OpenTelemetry, Prometheus + Grafana, Sentry, структурные логи (pino).

## 3. Event Ledger (ядро — реализовать первым)

- Таблица `audit_event`: `id, type, aggregateId, aggregateType, actorId, actorRole,
  payload(jsonb), refs(jsonb), createdAt`. **Только INSERT.** UPDATE/DELETE запрещены
  на уровне прав БД (отдельная роль без этих грантов) + триггер-защита.
- Любая мутация денег/склада/статуса пишет событие **в той же транзакции**, что и
  изменение данных (outbox-паттерн для доставки в шину).
- Исправления — **компенсирующим событием** (reverse), не правкой истории.
- Отчёты, Risk, Command Center, аналитика — **читатели** ledger → цифры не расходятся.

**Идемпотентность:** заказы — `Idempotency-Key` (header); платёжные webhook — dedup по
`txnId` (unique constraint). Повтор запроса не создаёт дубль.

---

## 4. Данные и целостность

- **Деньги — только целые (минорные единицы)**: храните тыйын/копейки `int`, не float.
  Все расчёты в целых, форматирование — на клиенте.
- **Транзакции + оптимистичные локи** (`version` поле) на Order, Inventory, CashShift.
- **Резерв склада** — атомарно: `UPDATE stock SET reserved=reserved+1 WHERE available>0`
  с проверкой; IMEI-товар — уникальный статус (нельзя продать дважды → 409).
- **Soft-delete** товаров (`archivedAt`), не физическое удаление.
- **Миграции** — версионные (Prisma migrate), обратимые, прогон на staging до prod.
- **Мультивалютность:** храните валюту + курс на момент операции; переоценка отдельно.

---

## 5. Безопасность

- **Auth:** JWT access (15 мин) + refresh (rotating, httpOnly), вход телефон+OTP/email/соцсети.
- **RBAC на сервере** — Role Permission Matrix проверяется в guard'ах, **клиенту не доверяем**.
- **2FA (TOTP)** обязательна для опасных действий и админ-ролей.
- **Approval-флоу:** опасное действие → `pending` + `approvalId`, исполняется только
  после решения уполномоченной роли (см. Approval Rules Matrix). Всё в аудите.
- **PII:** шифрование паспортов/телефонов (at-rest, pgcrypto/KMS), маскирование младшим
  ролям, лог доступа к PII, срок хранения, согласие (закон КР «О перс. данных»).
- **Прочее:** rate-limiting, CORS whitelist, CSP, валидация входа (Zod/class-validator),
  параметризованные запросы (Prisma), секреты в Vault/env, аудит входов + алерты на аномалии.
- **Бэкапы:** авто ежедневно + PITR (point-in-time recovery), регулярный тест восстановления.

---

## 6. Производительность и масштаб

- **Кэш:** Redis для каталога, цен, сессий; инвалидция по событиям.
- **Поиск:** Meilisearch/Elasticsearch, индексация по событиям товара (не прямой запрос в PG).
- **Пагинация:** cursor-based (не OFFSET) на больших списках.
- **N+1:** Prisma `include`/DataLoader; следить через APM.
- **Фото:** WebP/AVIF, ресайз на загрузке, CDN, ленивая загрузка, эконом-режим для слабой сети.
- **Очереди:** тяжёлое (уведомления, отчёты, AI, синк) — в фоновые воркеры (BullMQ).
- **Read-replica** PostgreSQL для отчётов/аналитики; горячая аналитика → ClickHouse.
- **Горизонтальное масштабирование:** stateless API за балансировщиком; sticky не нужен.

---

## 7. Офлайн (POS/курьер — критично)

- Локальная БД (Core Data/Room) — каталог, цены, остатки, очередь операций.
- Продажа офлайн → запись в локальную очередь → при сети **дельта-синхронизация**.
- **Разрешение конфликтов:** server-wins для остатков + компенсация; продажи не теряются.
- Идемпотентные операции (клиентский `opId`), чтобы синк не задваивал.
- Запрет опасных действий офлайн (возврат денег, большие скидки) — только онлайн.

---

## 8. Интеграции (через адаптеры, легко заменить)

- **Платежи:** MBank, O!Деньги, Bakai, эквайринг — webhook + idempotency + сверка.
- **Уведомления:** APNs/FCM (push), SMS-шлюз, Telegram Bot API, WhatsApp Business.
- **Карты:** 2ГИС/Google — зоны, маршруты (OSRM/GraphHopper для оптимизации).
- **AI:** LLM/Vision API за интерфейсом `AiProvider` (оценка Б/У, цены, ассистент) — заменяемо.
- **Оборудование:** сканер (HID/ML Kit), принтеры (ESC-POS/ZPL), терминал — через локальный агент.
- **Фискализация ГНС — вне системы** (ведёт владелец), не реализуем.

---

## 9. Тестирование и качество

- **Unit** (бизнес-правила, инварианты), **integration** (API+БД), **e2e** (Playwright/Detox).
- **QA-сценарии** из `AliStore QA Test Scenarios` = приёмочные тесты (P0 — блокирующие релиз):
  двойная продажа IMEI, оплата без резерва, касса не сходится, возврат без approval, офлайн-синк, дубль webhook.
- **Нагрузочное** (k6) на Orders/Payments/Search перед масштабированием.
- Code review обязателен; feature-флаги для рискованных фич; trunk-based + короткие ветки.

## 10. Порядок реализации (спринты)

1. Каркас монорепо + Prisma + миграции + сиды + CI.
2. **Event Ledger + Auth + RBAC** (фундамент).
3. Order state-machine (created→paid) + Payment + резерв IMEI → приёмочные тесты P0.
4. Web-витрина на реальном API (каталог→checkout) + профиль.
5. Нативный Клиент (iOS+Android): каталог, корзина, checkout, push, биометрия.
6. POS (офлайн-first, сканер, печать) → Cash Shift + Courier handover.
7. Возвраты/обмены/гарантия + Approval Inbox + Risk/Command Center (читатели ledger).
8. CRM/кампании/уведомления (consent) + Supplier/Закупки + HR/ЗП.
9. AI-слой по API. Инвентаризация, импорт данных, hardware-агент.
10. Нагрузочное, аудит безопасности, прогон QA → запуск MVP (Бишкек) → v1 → v2.

---

## 11. Definition of Done (для каждой фичи)
Серверная проверка прав/лимитов · пишет событие в Event Ledger · идемпотентна ·
покрыта тестами · есть состояния (loading/empty/error/success) · логируется ·
не ломает офлайн · обновлён API-контракт. Без этого — не в prod.
