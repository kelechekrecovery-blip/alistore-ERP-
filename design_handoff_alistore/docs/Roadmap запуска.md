# AliStore — Roadmap запуска (MVP → v1 → v2)

Что строить в каком порядке, чтобы запуститься быстро и безопасно. Опора: HANDOFF-доки, schema.prisma, api-and-events. Принцип — сначала то, что даёт продажи и защищает деньги.

---

## MVP (первый магазин, Бишкек) — «продавать и не терять деньги»
Цель: одна точка + сайт/приложение реально работают. Запуск за минимальный контур.

**Обязательно:**
- Ядро данных: Customer, Product/SKU, DeviceUnit(IMEI), Order, Payment, **AuditEvent (append-only)**.
- Каталог + карточка товара + корзина + checkout (сайт и/или приложение).
- Оплата: наличные, карта/терминал, MBank/O!Деньги QR; **оплата при получении**.
- POS 2.0: продажа, скидка (с лимитом), смешанная оплата, чек, offline-режим (очередь+синк).
- Склад: приёмка, остатки, резерв под заказ, продажа по IMEI (запрет двойной продажи).
- Смена кассы: открытие/закрытие + сверка наличных.
- Курьер: назначение → доставлен → сдача COD (Courier Cash Handover).
- Роли/доступ (базово) + вход по телефону+OTP.
- Гарантийный талон в приложении (привязка к IMEI).
- Скупка Б/У + договор (печать) — ваше ключевое преимущество.

**Осознанно НЕ в MVP:** франшиза, продвинутый CRM/кампании, AI-ценообразование, рассрочка-скоринг, мультифилиальные перемещения.

**Критерий готовности MVP:** заказ проходит created→paid→delivered→completed; деньги сходятся в Payment Ledger; ни один IMEI не продаётся дважды; касса и курьер сверяются; QA P0-сценарии (🔴) зелёные.

---

## v1 (сеть 2–4 точки) — «контроль и масштаб»
- Мультисклад + перемещения между филиалами.
- Approval Inbox + все опасные действия через approval (возврат, списание, цена, остаток, долг, удаление).
- Risk Center + Command Center (виджеты тревог).
- Возвраты/обмены полным циклом, Refund Money Flow, гарантийные кейсы с SLA.
- Долги и рассрочка + напоминания.
- KPI/зарплаты сотрудников, смены с фотоотчётом (Evidence Vault).
- Support Inbox (сайт/app/WhatsApp/Telegram), Notifications Outbox.
- Инвентаризация со сканером, Supplier RMA + Scorecard.
- Импорт данных из Excel/тетради.

**Критерий v1:** владелец видит всю сеть в одном окне; каждое опасное действие — с approval, evidence и в аудите; расхождения ловятся автоматически.

---

## v2 (лидерство + рост) — «AI и удержание»
- AI-слой: оценка Б/У по фото, динамические цены (разведка рынка), ассистент владельца, обогащение карточек, авто-категоризация.
- CRM: сегменты, кампании, ROI, лояльность/бонусы/уровни, реактивация.
- Telegram Mini App / WhatsApp-магазин по полному.
- Франшиза + аудит партнёрских точек.
- Омниканальность (click&collect), подарочные карты, страховка устройств, B2B/опт.
- Рекламный кабинет, маркетплейс сторонних продавцов.

**Критерий v2:** повторные продажи растут за счёт CRM/AI; сеть расширяется через франшизу на единой платформе.

---

## Сквозные требования (на всех фазах)
- **Event Ledger** — источник правды с первого дня; исправления компенсирующими событиями.
- **Серверная проверка** прав/лимитов/инвариантов (клиенту не доверяем).
- **Graceful degradation** сети и оборудования.
- **Consent** для маркетинга; PII-маскирование.
- **Оптимизация:** сжатие фото (WebP/AVIF), дельта-синк, эконом-режим.
- **Фискализация ГНС — вне системы** (ведёт владелец).

## Первые шаги разработки (неделя 1–2)
1. Поднять schema.prisma, миграции, сиды.
2. AuditEvent + сервис записи событий (обёртка для всех мутаций).
3. Order state-machine (created→paid) + Payment + резерв IMEI.
4. Минимальный POS-экран продажи (по прототипу POS 2.0) на реальном API.
5. Прогнать QA P0 «двойная продажа IMEI» и «оплата без резерва» как первые тесты.
