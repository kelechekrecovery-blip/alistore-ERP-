# AliStore — Инженерные решения (правила игры)

Этот документ — **источник правды по архитектуре**. Claude Code и любой разработчик
следуют ему. Если реализация расходится с этим файлом — сначала обсуждение (см. COLLAB.md),
потом код. Автор решений: дизайн/архитектура-ведущий (Claude Design).

---

## Зафиксированные решения (НЕ менять без обсуждения)

### 1. Архитектура: модульный монолит на старте
- **Решение:** один NestJS-бэкенд с чёткими модулями (Orders, Payments, Inventory,
  IMEI, Cash, Courier, Returns, Warranty, CRM, Notifications, Approval, AI).
- **Почему:** микросервисы на старте = преждевременная сложность, дорогой деплой,
  распределённые транзакции. Выносим в сервисы ТОЛЬКО горячие части под реальной нагрузкой.
- **Красный флаг:** если предлагается «сразу микросервисы / gRPC-меш» — стоп, обсуждаем.

### 2. Деньги — целые минорные единицы
- Хранить и считать в тыйынах (int64). НЕ float, НЕ строки. Формат — на клиенте.
- Любая денежная операция → запись в Payment Ledger + Event Ledger в одной транзакции.

### 3. Event Ledger — с первого дня, append-only
- `audit_event` только INSERT. Роль БД без UPDATE/DELETE грантов + защита триггером.
- Мутация данных и событие — одна транзакция (outbox для доставки в шину).
- Исправления — компенсирующим событием, не правкой истории.
- Не «добавим потом» — это фундамент, строится вторым после Auth.

### 4. RBAC и лимиты — только на сервере
- Клиентские проверки прав — косметика. Реальная проверка в guardّ ах по Role Matrix.
- Опасные действия → `pending` + approvalId, исполнение после решения (Approval Matrix).

### 5. Идемпотентность
- Заказы: header `Idempotency-Key`. Платёжные webhook: dedup по `txnId` (unique).
- Офлайн-операции: клиентский `opId`, чтобы синк не задваивал.

### 6. Склад / IMEI
- Резерв атомарно (условный UPDATE + version). IMEI нельзя продать дважды → 409.
- Soft-delete товаров (`archivedAt`). Серийный (IMEI) и штучный учёт — разные пути.

### 7. Офлайн (POS/курьер)
- Локальная БД + очередь операций + дельта-синхронизация.
- Конфликты: server-wins по остаткам + компенсация. Продажи НЕ теряются.
- Опасные действия офлайн запрещены (возврат денег, большие скидки).

### 8. Порядок разработки
- Сначала API + веб (ядро продаж), нативка (Swift/Kotlin) — от ГОТОВОГО API.
- НЕ строить нативку параллельно с нуля до готовности контрактов.

### 9. Стек — фиксирован
- Backend: NestJS + PostgreSQL + Prisma + Redis. Web: Next.js 14 + TS + Tailwind.
- iOS: Swift/SwiftUI. Android: Kotlin/Compose. Поиск: Meilisearch/Elasticsearch.
- НЕ менять на другой ORM/фреймворк без обсуждения.

### 10. Вне scope
- Фискализация ГНС — НЕ реализуем (владелец ведёт сам).

## Definition of Done (каждая фича)
Серверная проверка прав/лимитов · событие в Event Ledger · идемпотентность ·
тесты (вкл. QA P0) · состояния loading/empty/error/success · логирование ·
не ломает офлайн · обновлён API-контракт.
