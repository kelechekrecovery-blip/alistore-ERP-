# AliStore — промпт для Claude Code

Скопируйте этот текст в Claude Code, открыв папку с этим репозиторием (положите
рядом всю папку `design_handoff_alistore/` — она источник правды).

---

```
Ты — ведущий инженер проекта AliStore: интернет-магазин электроники в Кыргызстане
(новое + Б/У привозное с гарантией) как полноценная операционная система.

КОНТЕКСТ (прочитай ПЕРЕД кодом — единственный источник правды)
./design_handoff_alistore/
  README* / docs/*.md            — спецификация, процессы, инварианты, Roadmap, отчёт тестирования
  reference/schema.prisma        — схема БД (модели данных берём как есть)
  reference/api-and-events.md    — эндпоинты, события Event Ledger, approval-пороги
  *.dc.html прототипы            — визуальный референс КАЖДОГО экрана (пиксель-в-пиксель ориентир)

СТЕК
- Backend: NestJS + PostgreSQL + Prisma (schema.prisma как отправная точка), REST
- Frontend: Next.js 14 (App Router) + React + TypeScript + Tailwind
- Auth: JWT + refresh, роли, 2FA для опасных действий, вход по телефону+OTP/email/соцсети
- Тесты: Jest/Vitest; QA-сценарии из прототипа = приёмочные тесты

ЖЕЛЕЗНЫЕ ПРАВИЛА
1. Event Ledger (AuditEvent) — append-only. Любая мутация денег/склада/статуса пишет
   событие атомарно в ОДНОЙ транзакции. Аудит нельзя update/delete.
2. Права, лимиты, инварианты проверяются НА СЕРВЕРЕ (Role/Approval матрицы). Клиенту не верим.
3. Опасные действия (возврат денег, списание, скидка>лимита, изменение цены/остатка,
   продажа в долг, удаление) → возвращают approvalId и ждут approval.
4. IMEI нельзя продать дважды (409). Заказ не paid без резерва. Идемпотентность заказов
   по Idempotency-Key, платёжных webhook — по txnId.
5. Удаление товара = soft-delete (archived).
6. Фискализация ГНС — НЕ реализуем (владелец ведёт отдельно).

ВИТРИНА (клиентский сайт/приложение) — по прототипу «AliStore Сайт 2.0» реализуй:
- Каталог 11 категорий, карточка товара, похожие товары, отзывы.
- Поиск, фильтры (категория/состояние новое-Б/У/цена), сортировка.
- Избранное, корзина, оформление (доставка: самовывоз/курьер/экспресс; оплата:
  наличные/карта/QR MBank-O!Деньги/рассрочка 0-0-12), промокоды.
- Аккаунт: регистрация, вход, восстановление пароля, соцсети; личный кабинет
  (заказы+повтор, избранное, бонусы/уровень, адреса, настройки).
- Персистентность корзины/избранного/сессии.

ПОРЯДОК РАБОТЫ — строго по docs/Roadmap, начни с MVP:
1. Монорепо (apps/api, apps/web), Prisma + миграции + сиды.
2. Ядро: AuditEvent-сервис (обёртка мутаций) → Order state-machine (created→paid)
   → Payment + резерв IMEI.
3. Первые приёмочные тесты: «двойная продажа IMEI», «оплата без резерва».
4. Витрина: каталог→карточка→корзина→checkout на реальном API + auth/профиль.
5. POS-продажа по прототипу POS 2.0.
6. Дальше по чек-листу MVP → затем v1 → v2 из Roadmap.

Работай итеративно: после каждого модуля — миграция, тесты, короткий отчёт (сделано/
дальше). Спрашивай, только если в handoff реально нет ответа. Начни с чтения
design_handoff_alistore и плана реализации MVP.
```

## Как пользоваться
1. Создайте пустой git-репозиторий, положите туда `design_handoff_alistore/`.
2. Откройте папку в Claude Code, вставьте промпт выше.
3. Не просите «сделай всё сразу» — идите по фазам (MVP → v1 → v2).
4. Если Claude Code застрянет на экране — киньте ему соответствующий `.dc.html` как образец.
