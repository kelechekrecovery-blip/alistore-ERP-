СТЕК
- Backend: NestJS + PostgreSQL + Prisma (schema.prisma как отправная точка), REST + WebSocket (real-time статусы/склад)
- Web-витрина: Next.js 14 (App Router) + React + TypeScript + Tailwind (SEO, десктоп/моб. веб)
- Мобильные приложения — НАТИВНЫЕ (не PWA, не гибрид):
  · iOS — Swift + SwiftUI (min iOS 15)
  · Android — Kotlin + Jetpack Compose (min API 26)
  · 4 приложения: Клиент, Сотрудник, Курьер, POS. Общий backend/API.

  биометрия на устройстве (Face ID / отпечаток)
- Тесты: backend Jest/Vitest; iOS XCTest; Android JUnit/Espresso; QA-сценарии из прототипа = приёмочные тесты

НАТИВНЫЕ ТРЕБОВАНИЯ (обязательно задействовать железо — ради этого выбрана нативка)
- Камера/сканер штрихкодов и IMEI (POS, Сотрудник): AVFoundation / CameraX + ML Kit
- Push-уведомления: APNs (iOS) / FCM (Android) — заказы, доставка, гарантия, акции
- Офлайн-режим POS: локальная БД (Core Data / Room) + очередь операций + дельта-синхронизация при сети
- Печать чеков и этикеток: Bluetooth/ESC-POS принтеры
- Биометрия: LocalAuthentication / BiometricPrompt

- Каждое приложение — отдельный таргет в сторе (App Store / Google Play)

ЖЕЛЕЗНЫЕ ПРАВИЛА
1. Event Ledger (AuditEvent) — append-only. Любая мутация денег/склада/статуса пишет
   событие атомарно в ОДНОЙ транзакции. Аудит нельзя update/delete.
2. Права, лимиты, инварианты проверяются НА СЕРВЕРЕ (Role/Approval матрицы). Клиенту не верим.
3. Опасные действия (возврат денег, списание, скидка>лимита, изменение цены/остатка,
   продажа в долг, удаление) → возвращают approvalId и ждут approval.
4. IMEI нельзя продать дважды (409). Заказ не paid без резерва. Идемпотентность заказов
   по Idempotency-Key, платёжных webhook — по txnId.
5. Удаление товара = soft-delete (archived).
6. Фискализация ГНС — НЕ реализуем (владелец ведёт отдельно).

ВИТРИНА (клиентский сайт/приложение) — по прототипу «AliStore Сайт 2.0» реализуй:

- Поиск, фильтры (категория/состояние новое-Б/У/цена), сортировка.
- Избранное, корзина, оформление (доставка: самовывоз/курьер/экспресс; оплата:
  наличные/карта/QR MBank-O!Деньги/рассрочка 0-0-12), промокоды.
- Аккаунт: регистрация, вход, восстановление пароля, соцсети; личный кабинет
  (заказы+повтор, избранное, бонусы/уровень, адреса, настройки).
- Персистентность корзины/избранного/сессии.

ПОРЯДОК РАБОТЫ — строго по docs/Roadmap, начни с MVP:
1. Backend-монорепо (apps/api) + Prisma + миграции + сиды. Отдельные репо/папки:
   apps/api, apps/web (Next.js), apps/ios-* (Swift), apps/android-* (Kotlin).
2. Ядро API: AuditEvent-сервис (обёртка мутаций) → Order state-machine (created→paid)

3. Первые приёмочные тесты: «двойная продажа IMEI», «оплата без резерва».
4. Web-витрина: каталог→карточка→корзина→checkout на реальном API + auth/профиль.
5. Нативный Клиент (iOS Swift + Android Kotlin): каталог, карточка, корзина, checkout,
   push, биометрия — по прототипу «Клиент App 2.0». Экраны 1:1 с прототипом.
6. Нативный POS (офлайн-first, сканер, печать чека) — по прототипу «POS 2.0».
7. Нативные Сотрудник и Курьер (смены+фото, задачи, сканер IMEI, геотрекинг).

## Как пользоваться
1. Создайте пустой git-репозиторий, положите туда `design_handoff_alistore/`.
2. Откройте папку в Claude Code, вставьте промпт выше.
3. Не просите «сделай всё сразу» — идите по фазам (MVP → v1 → v2).
4. Если Claude Code застрянет на экране — киньте ему соответствующий `.dc.html` как образец.

## Важно про нативную разработку (Вариант B: Swift + Kotlin)
- **Backend и веб-витрина** строятся первыми — это общий фундамент для всех приложений.
  Один API обслуживает веб, iOS, Android, POS.
- **Каждое приложение — свой таргет**: iOS (Swift/SwiftUI) и Android (Kotlin/Compose)
  разрабатываются параллельно, но по одним и тем же прототипам и API-контрактам.
- Практично разбить на потоки: (1) API+web, (2) iOS-клиент, (3) Android-клиент,
  (4) POS, (5) Сотрудник+Курьер. Начинать всё от готового API.
- Прототипы `.dc.html` — это **дизайн-референс**, а не код: экраны воссоздаются
  нативными средствами (SwiftUI/Compose) пиксель-в-пиксель по ним.
- Дороже и дольше, чем один кросс-код, но даёт «родной» UX и максимальную
  надёжность офлайн/железа — критично для POS на кассе.
