## ЧАСТЬ 2. АРХИТЕКТУРА (как всё работает и на чём)
═══════════════════════════════════════════════════════════════════

### 2.1 Общая схема
```
КЛИЕНТЫ (точки входа)
 ├─ Веб-витрина      → Next.js 14 (SSR/SEO, десктоп+моб. веб)
 ├─ Клиент App       → Swift/SwiftUI (iOS) + Kotlin/Compose (Android)
 ├─ Сотрудник App    → нативно iOS/Android
 ├─ Курьер App       → нативно iOS/Android
 ├─ POS (касса)      → нативно, офлайн-first (планшет)
 ├─ ERP / Супер-админ→ веб (Next.js, в браузере)
 └─ Telegram-бот/Mini App → канал поверх того же API
        │  HTTPS / WSS
        ▼
 API GATEWAY  → авторизация, rate-limit, версии /v1, роутинг
        │
 BACKEND (NestJS, модульный монолит)
   модули: Auth, Orders, Payments, Inventory, IMEI, CashShift, Courier,
           Returns, Exchange, Warranty, Supplier, Purchasing, Pricing,
           Approval, CRM, Campaigns, Notifications, Support, HR, Analytics, AI
        │
   ┌────┴─────────────────────────────────────────────┐
   ▼                     ▼                ▼            ▼
 PostgreSQL           Redis          Meilisearch    S3/MinIO
 (основная БД)     (кэш/очереди/    (поиск по       (фото, чеки,
                    локи/сессии)     каталогу)       договоры, evidence)
        │
   EVENT BUS (Redis Streams / NATS) → EVENT LEDGER (append-only в PostgreSQL)
        │
   Фоновые воркеры (BullMQ): уведомления, отчёты, AI-задачи, синхронизация,
                              переоценка валют, авто-закупки, реактивация CRM
        │
   ВНЕШНИЕ ИНТЕГРАЦИИ (через адаптеры):
     платежи (MBank, O!Деньги, Bakai, эквайринг) · SMS-шлюз · APNs/FCM push ·
     Telegram/WhatsApp · карты 2ГИС/OSRM · LLM/Vision API · банк-скоринг рассрочки
```

### 2.2 Почему так
- **Модульный монолит**, не микросервисы: быстрый старт, простой деплой, обычные
  транзакции БД. Горячие части (Orders, Payments, Search, Notifications) выносим в
  отдельные сервисы ТОЛЬКО когда упрёмся в нагрузку. Границы модулей уже чёткие — вынести легко.
- **Event Ledger** — единый источник правды. Каждая мутация денег/склада/статуса пишет
  неизменяемое событие в той же транзакции. Отчёты/Risk/Command Center/аналитика читают оттуда
  → цифры никогда не расходятся. Исправления — компенсирующим событием.
- **Один API** обслуживает все клиенты (веб, iOS, Android, POS, Telegram). Нативка стартует
  от готового API, не параллельно с нуля.

### 2.4 Ключевые данные (см. reference/schema.prisma)
Customer, Product, ProductVariant, SKU, DeviceUnit(IMEI), Order, OrderItem, Payment,
CashShift, InventoryMovement, Reservation, Return, Exchange, WarrantyCase, TradeInDevice,
CourierRun, Delivery, Approval, AuditEvent, Campaign, SupportTicket, Supplier, PurchaseOrder,
Employee, Shift, Debt/Installment, GiftCard, Consent.

Правила данных:
- Деньги — **целые тыйыны (int64)**, не float.
- Транзакции + оптимистичные локи (`version`) на Order/Inventory/CashShift.
- Резерв атомарно; **IMEI нельзя продать дважды** (409).
- Soft-delete (`archivedAt`), не физическое удаление.
- Мультивалюта: валюта + курс на момент операции.

## ЧАСТЬ 3. ГЛАВНЫЕ БИЗНЕС-ПОТОКИ (что должно работать)
═══════════════════════════════════════════════════════════════════

1. **Заказ (онлайн):** каталог → корзина → checkout (доставка+оплата) → резерв IMEI →
   оплата (payment.received) → сборка → доставка/самовывоз → completed → гарантийный талон.
   Статусы: created→confirmed→reserved→paid→picking→packed→(courier/pickup)→delivered→completed.
2. **Продажа на POS:** скан/поиск → чек → скидка (approval если ниже лимита) →
   оплата (нал/карта/QR/рассрочка/смешанная) → чек → списание IMEI → KPI продавца.
3. **Оплата при получении (COD):** курьер доставил → cash handover в кассу → сверка.
4. **Возврат денег:** заявка → approval → возврат тем же способом → Payment Ledger → сверка.
5. **Обмен:** возврат старого + продажа нового атомарно + доплата + новая гарантия.
6. **Скупка Б/У:** AI-оценка (фото/описание) → проверка IMEI по базам → договор (данные
   клиента, паспорт) → выплата (касса) → оприходование → грейд → продажа.
7. **Гарантия/сервис:** обращение → диагностика → ремонт/замена/отказ (SLA-эскалация).
8. **Смена кассы:** открытие (остаток, чек-лист, фото) → продажи → закрытие (сверка,
   расхождение→инцидент, Z-отчёт). Передача смены между продавцами.
9. **Закупка:** AI-прогноз спроса → PO поставщику → приёмка по PO (сверка, пересортица) →
   оприходование + этикетки.
10. **CRM/удержание:** сегменты → кампания (consent-фильтр) → рассылка (Outbox) → ROI.
11. **Опасные действия** (скидка>лимита, возврат, списание, изменение цены/остатка,
    удаление, продажа в долг): → Approval → исполнение → Event Ledger + Evidence.
12. **Риски:** правила (двойной IMEI, частые возвраты, расхождения кассы, курьер не сдал,
    просрочка RMA) → Risk Center → Command Center владельца.

## ЧАСТЬ 4. БЕЗОПАСНОСТЬ (обязательно)
═══════════════════════════════════════════════════════════════════
- JWT access(15м) + refresh(rotating, httpOnly). Вход: телефон+OTP / email / соцсети / биометрия.
- **RBAC на сервере** (Role Permission Matrix), клиенту не доверяем. 2FA (TOTP) для опасных действий.
- Approval-флоу для опасных операций (пороги — в Approval Rules Matrix).
- PII (паспорт/телефон): шифрование at-rest, маскирование младшим ролям, лог доступа,
  срок хранения, согласие (закон КР «О персональных данных»).
- Rate-limit, CORS whitelist, CSP, валидация входа (Zod), секреты в Vault/env.
- Бэкапы ежедневно + PITR, регулярный тест восстановления. Аудит входов + алерты на аномалии.
- Идемпотентность: заказы (Idempotency-Key), webhook (txnId unique), офлайн (opId).

## ЧАСТЬ 5. ОФЛАЙН (POS/курьер — критично)
═══════════════════════════════════════════════════════════════════
Локальная БД (Core Data/Room) + очередь операций + дельта-синхронизация при сети.
Конфликты: server-wins по остаткам + компенсация, продажи не теряются. Опасные действия
офлайн запрещены. Каждая операция идемпотентна (opId).

## ЧАСТЬ 6. ПОРЯДОК РАБОТЫ (спринты — делай по очереди)
═══════════════════════════════════════════════════════════════════
1. Монорепо (`apps/api`, `apps/web`, `apps/ios-*`, `apps/android-*`), Prisma+миграции+сиды, CI, Docker.
2. **Event Ledger + Auth + RBAC** — фундамент. append-only с защитой на уровне БД.
3. Order state-machine (created→paid) + Payment + резерв IMEI.
4. Приёмочные тесты P0: «двойная продажа IMEI», «оплата без резерва» (из QA Test Scenarios).
5. Веб-витрина на реальном API: каталог→карточка→корзина→checkout + профиль/auth.
6. Нативный Клиент (iOS+Android): каталог, корзина, checkout, push, биометрия — по прототипу.
7. POS (офлайн-first, сканер, печать) + CashShift + Courier cash handover.
8. Возвраты/обмен/гарантия + Approval Inbox + Risk/Command Center (читатели ledger).
9. CRM/кампании/уведомления (consent) + Supplier/Закупки + HR/ЗП + расширенные модули.
10. AI-слой по API. Инвентаризация, импорт данных, hardware-агент. Нагрузочное + аудит → MVP (Бишкек).

## ЧАСТЬ 7. DEFINITION OF DONE (каждая фича)
═══════════════════════════════════════════════════════════════════
Серверная проверка прав/лимитов · пишет событие в Event Ledger · идемпотентна ·
покрыта тестами (вкл. QA P0) · есть состояния loading/empty/error/success · логируется ·
не ломает офлайн · обновлён API-контракт. Без этого — не в prod.
